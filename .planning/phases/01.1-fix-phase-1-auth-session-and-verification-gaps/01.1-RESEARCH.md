# Phase 01.1: Fix Phase 1 auth, session, and verification gaps - Research

**Researched:** 2026-03-27
**Domain:** Phase 1 remediation for auth security, session lifecycle correctness, and verification reliability
**Confidence:** HIGH

<phase_inputs>
## Phase Inputs

This phase exists because Phase 1 execution and post-execution review uncovered concrete defects that must be fixed before Phase 2 begins.

### Auth defects to remediate
- `JwtTokenService` falls back to a known default secret when `APP_JWT_SECRET` is unset.
- JWT validation does not enforce issuer equality with `collaborative-code-editor`.
- `JwtAuthenticationFilter` can throw on malformed `sub` claims instead of treating the token as invalid.
- `RefreshSessionService.rotate(...)` is race-prone and can mint multiple successor tokens from one refresh token under concurrent requests.
- Auth controller tests bypass the real security filter chain, so bearer-token enforcement is not actually verified.

### Session defects to remediate
- Owner transfer can leave multiple active `OWNER` participants after transfer and rejoin.
- Session join is not atomic, so cap enforcement and idempotent join semantics can break under concurrency.
- Cleanup can delete a room after a participant rejoins because it performs count-then-delete without an atomic guard.
- `leaveSession(...)` accepts already-left members and can extend the cleanup window indefinitely.
- Invite-code collisions are not retried and can surface as persistence failures.
- `JoinSessionRequest` validates only `@NotBlank`, so structurally invalid invite codes become `404` instead of `400`.

### Verification defects to remediate
- `./gradlew test` fails under the current default Java `25.0.1` runtime before project tests run.
- `FlywayMigrationTest` fails once Docker/Testcontainers wiring is correct because `coding_sessions.participant_cap` is `SMALLINT` in SQL but `int` in JPA.
- The full suite still depends on correct Docker socket wiring for Colima/Testcontainers.
- Current tests do not cover the most important concurrency and security regression paths.
</phase_inputs>

<research_summary>
## Summary

Phase 01.1 should be treated as a focused stabilization pass, not a redesign. The right goal is to harden the existing Phase 1 contracts while preserving the public API surface that Phase 2 expects: `/api/auth/*`, `/api/sessions*`, invite-code joins, owner transfer, and one-hour cleanup semantics all stay in place, but their implementation becomes safe under concurrency and honestly verifiable.

The safest implementation shape is:

1. **Repair verification and schema parity first enough to make failures trustworthy.**
   Align the entity/schema contract for `participant_cap`, keep the supported JDK/runtime explicit, and ensure Testcontainers uses a reproducible Docker configuration path.
2. **Harden auth next.**
   Fail fast on missing JWT secret, enforce issuer and subject parsing rules, and make refresh rotation single-use under concurrent requests.
3. **Harden session lifecycle after that.**
   Fix owner-role consistency, serialize join/cap changes, make cleanup conditional, and close the malformed invite-code/input-contract gaps.
4. **Finish with regression tests that hit the real behavior being fixed.**
   This phase should not claim success from Mockito-only tests; it needs focused security and concurrency coverage plus a full suite that is green on a supported runtime.

**Primary recommendation:** plan this phase as three execution plans: verification/schema repair, auth hardening, and session/concurrency hardening plus regression coverage.
</research_summary>

<recommended_patterns>
## Recommended Patterns

### Pattern 1: Fail-fast JWT configuration and strict token validation
**What:** Remove the static default JWT secret, require an explicit secret in configuration, and reject tokens unless both signature and issuer match expected values.
**When to use:** Any self-issued JWT flow where the app is the sole issuer of bearer tokens.
**Why here:** The current implementation can silently run in an insecure configuration and accepts tokens missing issuer validation.
**Concrete guidance:**
- Bind the signing secret through validated configuration rather than a permissive `@Value` default.
- Reject startup when the secret is blank or too short.
- Require issuer `collaborative-code-editor` during parsing.
- Guard `UUID.fromString(...)` and treat malformed `sub` as authentication failure, not as a server error.

### Pattern 2: Atomic refresh-token rotation
**What:** Convert refresh rotation from a read-check-write flow into a transaction that guarantees only one request can consume a given refresh session.
**When to use:** Single-use rotating refresh tokens where reuse detection matters.
**Why here:** The current `findByTokenHash(...) -> check revokedAt -> save new -> revoke old` flow can be executed concurrently.
**Concrete guidance:**
- Use a repository method that locks the refresh-session row (`PESSIMISTIC_WRITE`) before evaluating `revokedAt`, or use a conditional update pattern where only one caller can transition the row from active to revoked.
- Preserve the existing `device_id` and `replaced_by_session_id` chain.
- Keep reuse handling scoped to the affected device or session chain, but make the first winner deterministic and every loser a reuse path.

### Pattern 3: Session lifecycle mutations must be serialized per session
**What:** Join, leave, owner transfer, cleanup-window updates, and cleanup deletion should all operate under a per-session transaction boundary that prevents stale count-based decisions.
**When to use:** Collaboration/session models with caps, ownership, and TTL semantics.
**Why here:** The current session logic is correct only in single-threaded execution.
**Concrete guidance:**
- Lock the `coding_sessions` row during join/leave/cleanup decisions, or use conditional repository updates/deletes that encode the guard in SQL.
- Demote the old owner role when ownership transfers.
- On rejoin, set role based on `owner_user_id`; do not blindly reactivate a stale `OWNER` row.
- Reject `leaveSession(...)` when the participant is already `LEFT`.
- Retry invite-code generation on unique-key conflict with a bounded attempt count.

### Pattern 4: Prefer schema/entity alignment over rewriting history
**What:** Fix the current schema-validation mismatch by aligning the entity mapping to the existing column type unless a migration is clearly required.
**When to use:** Post-execution repair phases where the schema already exists and the logical domain still fits the stored type.
**Why here:** `participant_cap` is already `SMALLINT` in Flyway and only needs to store values in the low tens.
**Concrete guidance:**
- Change the entity field to `short`/`Short`, or explicitly map the existing `SMALLINT` type.
- Avoid rewriting V1 to `INTEGER` unless there is a strong reason to widen the column now.

### Pattern 5: Test the real behavior that broke, not just mocked flows
**What:** Add coverage at the layer where the defect exists.
**When to use:** Remediation phases driven by actual bug findings.
**Why here:** Existing controller tests mostly validate DTO and status plumbing, not security chain behavior or mutation races.
**Concrete guidance:**
- Add tests for missing/invalid/valid bearer tokens against the real `SecurityConfig` + `JwtAuthenticationFilter`.
- Add service/repository tests for owner rejoin, repeated leave rejection, invite-code validation, and collision retry.
- Add concurrency-oriented tests around refresh rotation and session join if practical at the service/repository boundary.
- Re-run full migration tests under the supported Docker/JDK setup before declaring the phase complete.
</recommended_patterns>

<common_pitfalls>
## Common Pitfalls

### Pitfall 1: Fixing auth logic without fixing the verification path
**What goes wrong:** The implementation becomes safer, but the test suite still bypasses the real filter chain and never proves it.
**How to avoid:** Add at least one test slice or integration test that exercises `SecurityConfig` and `JwtAuthenticationFilter` directly.

### Pitfall 2: Treating race conditions as “rare enough”
**What goes wrong:** Owner transfer, join idempotency, and refresh rotation still fail under concurrent requests in production even though unit tests are green.
**How to avoid:** Move critical guards into locked queries, conditional updates, or conditional deletes.

### Pitfall 3: Closing the Testcontainers failure without closing schema drift
**What goes wrong:** Docker wiring works, but the suite still fails because Hibernate validation disagrees with Flyway.
**How to avoid:** Resolve the `participant_cap` type mismatch first, then use the migration test as the gate for verification.

### Pitfall 4: Expanding this into a redesign
**What goes wrong:** The remediation phase starts inventing new auth/session abstractions and drifts into Phase 2 work.
**How to avoid:** Preserve the existing API contracts and public behavior; fix correctness and verification only.
</common_pitfalls>

## Validation Architecture

Phase 01.1 should be verified in layers:

| Layer | Purpose | Command |
|------|---------|---------|
| Auth/session unit + slice | Fast feedback for hardening changes | `JAVA_HOME=<supported-jdk> ./gradlew test --tests "com.collabeditor.auth.*" --tests "com.collabeditor.session.*"` |
| Migration/integration verification | Prove Flyway + JPA + Docker/Testcontainers actually work | `JAVA_HOME=<supported-jdk> DOCKER_HOST=<docker-socket> ./gradlew test --tests "*FlywayMigrationTest"` |
| Full phase regression | Prove Phase 1 remains green after fixes | `JAVA_HOME=<supported-jdk> DOCKER_HOST=<docker-socket> ./gradlew test` |

**Recommended validation emphasis:**
- One plan should explicitly make the full suite trustworthy again.
- Every auth hardening change should have a matching negative-path test.
- Every session race or state-consistency fix should have a regression test for the exact broken path.
- Final verification should use a supported JDK (Corretto 24 or Java 17 toolchain path) and a working Docker socket configuration.

**Manual-only checks that still matter:**
- Live bearer-token flow against a running app and database.
- Live session create/join/leave/cleanup flow with two users after the fixes land.
