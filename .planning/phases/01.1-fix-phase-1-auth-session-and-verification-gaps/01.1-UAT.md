## Tests

### 1. Fail-Closed JWT Startup
expected: Start the backend with `APP_JWT_SECRET` unset or shorter than 32 characters. The application should fail during startup with a configuration error about `app.security.jwt-secret` instead of booting with an insecure default.
result: pass

### 2. Protected Sessions Route Requires Authentication
expected: Start the backend with a valid 32+ character `APP_JWT_SECRET`, then call `GET /api/sessions` without an `Authorization` header. The API should return HTTP 401 and no session payload.
result: pass

### 3. Login Produces Working Access and Refresh Credentials
expected: Register or log in with a valid user. `POST /api/auth/login` should return an access-token payload and set the `ccd_refresh_token` cookie; using that bearer token on `GET /api/sessions` should return HTTP 200 with JSON.
result: pass

### 4. Invalid Bearer Tokens Are Rejected Cleanly
expected: Send `GET /api/sessions` with a malformed or tampered bearer token. The API should return HTTP 401 instead of a 500 or framework error page.
result: pass

### 5. Refresh Rotation Is Single-Use
expected: After login, send two refresh requests close together with the same `ccd_refresh_token` cookie. Exactly one request should succeed with HTTP 200 and a new token pair; the other should be rejected with HTTP 401 for invalid/reused refresh state.
result: pass

### 6. Malformed Invite Codes Fail at the HTTP Boundary
expected: Send `POST /api/sessions/join` with an invite code like `bad-code`. The API should return HTTP 400 rather than treating it like a missing session.
result: issue
reported: "POST /api/sessions/join returned HTTP 401 for inviteCode bad-code, including after logging in again for a fresh TOKEN."
severity: major

### 7. Lowercase Invite Codes Still Join the Intended Session
expected: Create a session, then join it from another authenticated user using the lowercase form of the returned invite code. The join should succeed with HTTP 200 and identify the same session/invite code in canonical uppercase form.
result: pass

### 8. Owner Leave and Rejoin Preserve One Coherent Owner
expected: With at least two participants in a session, have the owner leave and then rejoin. Exactly one remaining participant should become owner when the owner leaves, and the returning original owner should rejoin as a participant unless ownership was still assigned to them.
result: pass
